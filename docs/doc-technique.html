Format des données Grist

Grist retourne les tables au format columnar (par colonne, pas par ligne) :

// Exemple : table avec 3 lignes
{
  id:  [1, 2, 3],
  Nom: ["Paris", "Lyon", "Marseille"],
  Pop: [2161000, 513000, 861000],
}

// ChoiceList (ex: sélections multiples)
Selection: [["L", "Fixe", "Ciblée"], ["L", "Rotation"], null]

// Ref:* (référence vers une autre table)
Commune: [12, 7, 34]   // entiers = rowId de la table cible

// Dates
Debut: [1704067200, 1706745600]  // Unix secondes (pas ms !)
Attention : Unix secondes Grist stocke les dates en secondes Unix, pas en millisecondes. Utiliser unixSecondsToISODate() de meta.ts pour la conversion.

Lib Grist

src/lib/grist/

init.ts — Détection du contexte

Point d'entrée partagé par tous les widgets. Expose :

  • initGristOrMock() — détecte le mode et initialise grist.docApi ou son équivalent
  • getMode() — retourne le mode actif : "grist" | "mock" | "rest" | "none"
Détection iframe La détection isInFrame vérifie window.self !== window.top. En dehors d'une iframe (accès direct à l'URL), le mode "grist" ne s'active pas même si window.grist existe.

hooks.ts — Hook principal

useGristInit(opts) est appelé au montage de chaque widget. Il orchestre l'initialisation complète :

useGristInit({ requiredAccess: "full" }) │ ├─ 1. Charge grist-plugin-api.js si nécessaire (CDN docs.getgrist.com) ├─ 2. Appelle initGristOrMock() ├─ 3. En mode "grist" : appelle grist.ready() avec requiredAccess │ └─ 4. Récupère l'utilisateur connecté : ├─ grist.getUserProfile() # postMessage, Grist ≥ 1.1.5 └─ GET n8n proxy (fallback) # CORS proxy via webhook Retourne : { mode, docApi, gristUser }
OptionTypeDescription
requiredAccess"read table" | "full"Niveau d'accès demandé à Grist. "full" requis pour écrire.

meta.ts — Types et utilitaires

Types principaux

interface GristDocAPI {
  fetchTable(name: string): Promise<Record<string, unknown[]>>;
  applyUserActions(actions: unknown[][]): Promise<unknown>;
  getAccessToken(opts?): Promise<{ token: string }>;
  getUserProfile?(): Promise<GristUser>;
}

interface RefCache {
  [colId: string]: { id: number; value: string }[];
}

Résolution de colonnes

loadColumnsMetaFor(docApi, tableName) charge les métadonnées via la table système _grist_Tables_column. Utile pour résoudre dynamiquement les noms de colonnes sans les hardcoder.

ensureRefCache(docApi, table, col) construit un cache des valeurs de colonnes Ref:* pour affichage dans les dropdowns.

rest.ts — Client REST via n8n

Implémente la même interface que grist.docApi mais via des appels HTTP au proxy n8n (NEXT_PUBLIC_GRIST_PROXY_URL).

Widget (browser) n8n Proxy Grist REST API │ │ │ │ GET ?table=COMMUNES │ │ │ ──────────────────► │ │ │ │ GET /api/docs/{id}/tables │ │ │ ──────────────────────────► │ │ │ { records: [...] } │ │ │ ◄────────────────────────── │ │ { records: [...] } │ │ │ ◄────────────────── │ │
Trick CORS Les requêtes POST utilisent Content-Type: text/plain au lieu de application/json. C'est une simple CORS request qui ne déclenche pas de preflight OPTIONS — contournement volontaire des restrictions CORS.

Flux de saisie

Sélection commune │ ▼ loadDecompteForCommune() │ fetchTable("DECOMPTE") filtré communeId + mois + année │ → peuple counts + savedCounts + decompteId ▼ Utilisateur modifie les compteurs │ ▼ handleSave() │ ├─ decompteId existe ? │ ├─ OUI → UpdateRecord DECOMPTE [decompteId] │ └─ NON → AddRecord DECOMPTE (crée la ligne) │ └─ AddRecord DECOMPTE_LOGS (Δ pour chaque type modifié)

Journal (DECOMPTE_LOGS)

À chaque sauvegarde, autant de logs que de types modifiés. Chaque log enregistre :

  • Delta — différence (positif = hausse, négatif = baisse)
  • Type — code de l'acte ("PC", "DP"…)
  • Timestamp — ISO 8601
  • CommuneNom — dénormalisé pour la lisibilité

Le bouton Annuler dans la sidebar applique le delta inverse via UpdateRecord.

Recherche commune (scoreCommune)

Scoring de pertinence pour l'autocomplete. Le comportement diffère selon que la requête ressemble à un code INSEE (chiffres) ou à un nom :

// Recherche par NOM
score 0  → nom exact
score 1  → nom commence par la requête
score 2  → nom contient la requête
score 3  → INSEE commence par les chiffres
score 4  → INSEE contient les chiffres
score 99 → pas de match → exclu

// Recherche par INSEE
score 0  → INSEE exact
score 1  → INSEE commence par la requête
score 2  → nom commence par la requête
score 3  → nom contient la requête
score 4  → INSEE contient la requête
score 99 → pas de match → exclu

Maximum MAX_COMMUNE_RESULTS (25) résultats retournés, triés par score croissant.

Logique de filtrage par période

Une ligne Communes_Statut est visible si elle chevauche la période affichée :

Période affichée :  [start ─────────────────── end]
Ligne incluse si :  row.debut <= end  ET  row.fin >= start

Cas particuliers :
  - row.debut = null → pas de borne de début (toujours "en cours" depuis avant)
  - row.fin   = null → pas de borne de fin (encore actif)

Combiné avec deux filtres supplémentaires :

  • tagFilters — au moins un tag de la ligne doit être dans l'ensemble actif
  • arrFilters — l'arrondissement de la commune doit être sélectionné

Identité utilisateur

L'affichage du nom de l'utilisateur connecté dans la top bar suit cette cascade :

1. grist.getUserProfile() → postMessage, Grist ≥ 1.1.5 ↓ (échec) 2. getAccessToken() + GET n8n → serveur n8n appelle /api/profile ↓ (échec ou proxy non configuré) 3. gristUser = null → rien affiché dans la top bar
Configuration du proxy n8n Pour activer le fallback utilisateur, configurer le GitHub Secret GRIST_PROFILE_PROXY_URL avec l'URL du webhook n8n correspondant.
